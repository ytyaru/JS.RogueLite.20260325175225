# 画面設計書

## 基本

項目|最小|最大|理由
----|----|----|----
画面サイズ|`240x320`|`1920x1080`|スマホ/PC
小字サイズ|`12px`|`16px`|非漢字／漢字可視最小サイズ
大字サイズ|`18px`|`24px`|16pxより大きく差が視認できそうな最小／画面サイズ/10字
行間|`0em`|`1.5em`|非文章など行間不要時／日本語の文章で読みやすい最小サイズ
字間|`0em`|`0.05em`|非文章など字間不要時／日本語の文章で読みやすい最小サイズ

* `font-size:16px`: 縦16px。横は全角16px、半角8px。（等幅時）
* 対応HID: マウス、キーボード
* フォーカスループ: ブラウザのUIにフォーカスせずDOM内要素間だけでフォーカスすること

### 2-1. 全体

```
Hero  1  Enemy
10/10    10/10
札名 札名 札名
効果 効果 効果
 🄹   🄺   🄻
説明
```

各項目は以下の通り。

```
   自名    T     相名
Life/Body     Life/Body
札名     札名     札名
効果     効果     効果
キー
説明
```

自名と相名は`Hero`/`Enemy`など固定値を使う。左がプレイヤー、右がNPCであり、それぞれの生命点が以下に続く。その所有関係が伝わるように表示すること。

Life/Bodyは札の説明文に登場する用語。説明文を理解するためには、画面のこの部分に表示された数`10/10`が、それぞれ`Life/Body`(`生命点/身体点`)という名前の異なる項目であることを伝える必要がある。たとえば説明文の中にある各用語にフォーカスが当たるようにし、当たった時は数と字が交互に入れ替わる表示にすることで、どこに表示された数が何という名前であるかを示すようにするなど、画面の見せ方で理解できるような工夫をすべき。

札はボタンであり押すと実行される。札は名前と効果に大別して表示される。名前は札の種別を一目で区別するための文字列である。効果はその要約である。効果は可変な状態を表示することが絶対条件であり、ついでに効果詳細を察するか思い出すための内容にできれば尚良し。それ以上のことは説明を見る必要がある。ここはパッとみてサクッと意思決定するために必要な最小限の情報だけを表示すべき。

キーはキーボードのキーを押したらその上の札が選択されるキーを表示する。キーに関してはスマホの場合省略すべき。但しPCかスマホかを判別する方法はない。画面サイズから推測するしかないが、キーボードのない大画面のタブレットだと判定できなくなる可能性がある。また、BlueToothでキー接続したスマホの場合、キーボードのキーを表示すべきかどうか、技術的に可能かどうかも考慮し決定する必要がある。

説明はフォーカスが当たっている札の詳細説明を表示する。ゲームルールの根幹であり重要。その説明文は可視性があるべきであり、読んで理解できる文章であるべき。専門用語が登場するので、それが画面上にあるどの項目を指しているのかはアニメーションなど言外の要素によってプレイヤーに通知する工夫を要する。説明は一度覚えたら不要のため画面の下端に表示する。覚えた後は札ボタンの名前や効果だけ見れば済むだろう。

### 2-3. 札

```
札名
------------------
効果トークン<wbr>
...
効果トークン<wbr>
```

札は名前と効果に分かれている。その区分には下線を引いたほうが良いかもしれない。

効果は効果トークンの配列で構成されている。効果トークンには次のタイプがある。

1. 変数操作
2. 特性表記

#### 2-3-1. 変数操作

変数操作の効果トークンは、`相命-3`のような表記になる。

項目|例|意味
----|--|----
所有者|`相`/`自`|変数の所有者
変数|`命`/`体`|変数の種別
操作|`+`/`-`|変数の操作種別
量|`3`|操作する変数の量

* この所有者の、この変数を、このように操作します。その量はいくつです。

上記のような文章で説明できることを端的に表示した効果トークンである。

#### 2-3-2. 特性表記

特性表記の効果トークンは、`[連斬]`のような表記になる。

`[短縮特性名]`という表記になる。特性名の短縮形を`[]`で囲む書式だ。

本作には以下二種類のみ存在する。

* `[連斬]`
* `[自動]`

#### 2-3-3. 表示

札の表示は以下のようになる。

```
      斬る
----------------
相♥-1♡-3[連斬]
```

名前は大きめに表示したいが、難しいなら効果と同じでも良い。名前と効果の間にはボタンの枠線と同色の横線があると仕切りが可視化されて理解しやすくなるだろう。

画面サイズごとの最適化を考慮すべきだ。画面サイズが240x320でありボタンを横に3つ並べると、ボタン1個あたりの横幅は240/3=80pxと小さい。効果トークンの字サイズを12pxと仮定しても、80/12=6.6、6字しか入らない。

そこで改行する。縦幅は320pxであり余裕があるからだ。通常は字が横の端に到達すると自動折返しされるが、効果トークン途中で自動折返しされると読みづらくなる。そこで効果トークン内は自動折返し禁止にしつつ、トークン間に`<wbr>`を配置することで読みやすい位置で改行されるよう工夫する。たとえば以下のようになる。

```
斬る
------
相♥-1♡-3
[連斬]
```
```
斬る
------
相♥-1
　♡-3
[連斬]
```

`♡-3`には所有者の`相`がなく省略されている。これは前にある効果トークンと同じ所有者の場合は省略することで字数を減らすというルールにしてあるためだ。しかしそのため字の位置がズレて読みづらくなる。これを何とかするには右寄せが簡単だが、ボタン内全体としては中央寄せにしたい。するとHTML構造が更に複雑化してしまうだろう。

読みやすさのために、実装をどこまで複雑にするか。字種やサイズの組合せをどうするか。考慮すべきことが山ほどある。

* 12px英語: 字が潰れることなく読める: `OL-1 B-3 [Combo]`
* 16px英語: 字が潰れることなく読める: `OL-1 B-3 [Combo]`
* 12px日本語: `[連斬]`などの字が潰れる: `相命-1体-3[連斬]`
* 16px日本語: 行数が増える。`相`が略されている時に位置がずれる: `相命-1体-3[連斬]`
* 12px絵文字: `👿♥-1♡-3🗡️`
* 16px絵文字: `👿♥-1♡-3🗡️`
* 12px絵字日: `相♥-1♡-3[連斬]`
* 16px絵字日: `相♥-1♡-3[連斬]`
* 12px絵字英: `O♥-1♡-3[Combo]`
* 16px絵字英: `O♥-1♡-3[Combo]`

札名を絵文字で表示すれば名前や特性の表記は不要かもしれない。

ボタンの高さを作中最大のものに統一するならば、画面・文字サイズ、言語における全パターンを網羅し、最大になるサイズを算出し、それに合わせることになる。

ボタンの高さを指定せず自動にするなら、表示内容ごとに違うサイズになってしまい、見た目がガタガタになって汚くなる。

結局、高さをいくつにすべきかは確定できず。

### 2-4. 結果ダイアログ

```
結果
------------
勝利！
敗北…
  [OK]
```

JSの`alert()`関数で結果表示する。この時ダイアログのタイトルは変えられない。本当は`結果`にしたかったが。表示位置も制御できない。各ブラウザの仕様に従うため統一できない。メタ目線になるためゲームへの没入感が失せる。

数々の問題がありながら、それでも`alert()`を使う理由は、実装簡略化のためだ。`<dialog>`を使えば互換性問題が、`<div>`を使えば実装難度の問題がある。作り込むコード量が一気に増えるため今回は`alert()`で実装する。

`OK`ボタンを押下すると開始画面へ戻る。
