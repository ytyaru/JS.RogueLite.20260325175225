# 札説明文規則

札は本ゲームの根幹である。その札にどんな効果があるかをどう伝えるかがゲーム性に直結する。よって本文書はプレイヤにどう札の効果を伝えるか考察し、その伝達方法を定義する。

## 前提

本ゲームは画面構成により次のようなことを言外に伝える。

* 本ゲームは対戦ゲームである
* 本ゲームはプレイヤが勝利を目指すものである
* 本ゲームは先に相手の生命点を0にしたほうの勝利である

上記を前提として、以下も同様に伝える。

* 本ゲームは意思決定ゲームである
* 本ゲームはボタン押下によって意思決定する
* ホタンは3つある
* 3つあるボタンにはそれぞれ効果がある
* 3つあるボタンの中から一つ選ぶ

上記を前提として、以下も同様に伝える。

* 効果は識別できる
* 効果は説明される
* 効果は理解できる

## 伝達

プレイヤは以下の戦闘画面を見て、自分と相手の生命点を知り、次の手を選ぶ。

```
自名                  相名
10/10                10/10
+------+ +------+ +------+
|札名１| |札名２| |札名３|
|状態  | |状態  | |状態  |
+------+ +------+ +------+
+------------------------+
|フォーカスの当たっている|
|札の説明文を            |
|ここに表示する          |
+------------------------+
```

1. 札の種別を識別する
2. 札の状態を識別する
3. 札の効果を理解する

プレイヤはまず札の効果を理解する必要がある。これは自然言語における文章で説明する。この時、用語や文言は表記ゆれを一切せぬよう完全に統一された記述をすべきである。また、自明の内容は省略し、必要最小限の文章にすべきだ。

次に札の状態を識別する。札の中には状態が変化するものも存在する。状態によって効果が変わるため、状態を迅速に識別する必要がある。故に状態は説明よりも先に見やすい上部に配置する。

次に札の種別を識別する。札の効果を覚えたら、札の種別を一瞬で識別することで迅速に意思決定し、軽快にゲームをプレイできる。この時に重要なのは一目で分かることだ。ダラダラと長い説明文を読まねば識別不能だと軽快さが失われる。そこで一目見て識別できる工夫が必要だ。

### 1. 札の種別を識別する

札の種別を識別する方法は以下。

1. アイコンを用意する
2. 絵文字を用意する
3. 名前を用意する

#### 1. アイコンを用意する

これが最善策だ。札ごとに一目で識別できるアイコンを用意する。これは様々な観点から適切だ。端的に識別するには最高の方法である。また、将来スマホ対応した時にタッチ操作をする際、指の大きさを考慮して48pxサイズは確保すべきである。この時、その矩形内に最大限の識別しやすい記号を1つだけ表示することが、スマホという小さな画面において最も画面を効率よく使用した情報伝達と言えるからだ。

アイコンは1:1なり4:3なりのSVGなりPNGなりの画像で用意する。レスポンシブ対応を考えればSVGであるべきだ。また、ダークモード／ライトモードの切替を考慮すると背景は透明色で、単色の図形として設計すべきである。

問題は読み方が不明であることだ。頭の中でこの札を呼ぶ際、その呼び名が存在しないと識別しづらい。よって別途、名前が必要である。名前はアイコンの下に表示することになるだろう。その際、アイコンの上に重ねるのか、それとも下に追記する形になるのかは画面サイズ次第だろう。また、1:1か4:3などアイコンのアス比も考慮すべきだ。

札の効果が変わる状態に関しては、状態の名前や量をアイコンの右上に重ねて小さく表示するという工夫が考えられる。

最大の問題は、開発コストが大きいことだ。札の種類が増えるたび専用アイコンを作成せねばならない。そのアイコン図形は識別可能であるべきだ。他の図形と形が似ていないことを保証せねばならない。それでいながら、その図形から効果を連想できるものであるべきだ。アイコン作成は新たな字を作成することに等しい重労働である。

「実装量を減らす」という開発方針に合わない。最低限プレイヤに意味が伝われば良い。そのためにはアイコンが無くとも名前さえあれば済むはずだ。よってアイコン作成はしない。

#### 2. 絵文字を用意する

これは次善策だ。アイコン作成が困難ならば絵文字で代用すれば良い。但し問題はプレイヤが絵文字フォントを用意せねばならないことだ。環境によっては絵文字フォントをインストールすることができない場合もあるだろう。

また、絵文字はフォントや字種によって表示サイズがバラバラになることも問題だ。半角・全角よりも更に統一性のない状態になってしまい、小さな画面内で綿密に設計せねばならぬ状況には不適切な選択だ。

#### 3. 名前を用意する

最も再現性の高い方法だ。札ごとに異なる名前をつけて識別可能にする。等幅フォントを使えばサイズ不均等問題も解決する。特殊なフォントも不要だ。母国語と英語のフォントくらいはどの環境にも入っているだろう。但し、言語によって異なる文字列にせねばならない。すなわち国際化`i18n`の対応が必要になる。

また、限られた画面サイズ内に収まる文字数で名付けねばならない。例えば240x320だと4字になる。横幅240px内で3つのボタンを横並びに配置するため、1ボタンあたり80pxとなる。漢字を識別するために必要な最小`font-size`は`16px`だ。80/16px=5字である。もしここにボタンの境界線、`padding`、`margin`、`letter-spacing`が僅かでもあるなら4字になる。OSやブラウザの描画、フォントファミリによる誤差なども考慮すると、4字と考えておくべきだ。

本ゲームでは以下のような名前の札を用意した。`自然治癒`が最長である。私の開発環境Firefox142ではギリギリに見える。5字は入らなそうだ。

絵|日|英
--|--|--
`🧪`|`薬`|`Medicine`
`⚔️`|`斬る`|`Slash`
`💥`|`突貫`|`Rush`
`✨`|`自然治癒`|`AutoHeal`

英語は8字入るだろう。ギリギリが2つある。また、SlashとRushは響きが似ており音による識別にやや難があるか？　尤も、今回の開発フェーズでは国際化を切り捨てて日本語のみに絞っているため考慮不要である。但し将来は対応したいため命名くらいは考察しておきたい。

### 2. 札の状態を識別する

`斬る`の札には状態がある。身体点を減らす値が0〜2の整数幅だけ加算される。これを最小限で識別可能にするために`斬る+1`のように名付ける。

`斬る`, `斬る+1`, `斬る+2`の3つの名前に変化する。これにて状態の区別を可能としつつ、4字(1字/16px)≦80pxの制約内に収まるようにした。

尤も、名前と状態だけでは、まだ札の効果は理解できないが。

### 3. 札の効果を理解する

名前や状態の最小表示だけでは札の効果を理解できない。そこで、次の二段階で効果説明することを考えた。

方法|伝達方法
----|--------
効果要約|札ボタン内で名前の下に効果の要約を記号等で短縮記述する。
効果詳細|札ボタン外の下部で自然言語により詳細を記述する。

結論から言えば、効果要約は実装しない。札ボタンには名前と状態のみを記述し、それ以外の説明は詳細で記述する。これは開発方針の「実装量を減らす」を鑑みた決定である。

#### 効果要約

名|絵|日|英
--|--|--|--
`🧪薬`|`😀♥+3`|`自命+3`|`SL+3`
`⚔️斬る`|`👿♥-1♡-1`|`相命-1体-1`|`OL-1 B-1`
`💥突貫`|`👿♥-3😀♥-1`|`相命-3自命-1`|`OL-3 SB-1`
`✨自然治癒`|`😀♥+1`|`自命+1`|`SL+1`

効果要約には主要な効果のみを短縮記号で示す。世界共通言語として絵文字と数学記号とアラビア数字だけを使う。それ以外は自然言語として各言語を使う。すると主に上記の3通りある。

効果要約は札ボタン内に表記するため、限られたスペース内に収めねばならない。故に字数は限られている。すると上記のような表記くらいしか許されない。それでも1行内に収まらず、複数行にならざるを得ない。

1ボタン80pxで16px/1字とすると4字/1行しか入らない。すると`👿♥-1♡-1`や`👿♥-3😀♥-1`は2行に分けねばならない。しかし中途半端な所で改行されると視認性が下がってしまう。

##### 効果要約トークン

効果要約を、複数の効果要約トークンに分割する。そしてトークン内は改行禁止にする。トークン間のみ改行許可することで、読みやすい位置での改行にする。ようするに`👿♥-3😀♥-1`を`👿♥-3`と`😀♥-1`に分けて必要に応じて改行する。

* 札
    * 効果
        * 要約
            * トークン
                * 変数操作
                    * 主体(属性の所有者)
                    * 属性(生命点／身体点)
                    * 操作(減らす／増やす)
                    * 量(1,2,3等)
                * 特性表記
                    * 名称

札は効果を発揮するものであり、その要約はボタン内に表示する。その場合、要約はトークンに分けられる。トークンには変数操作系と特性表記系の2種類ある。このうち特性表記は省略しても良い。

これを実装する場合、以下のような複雑な構造化を要する。

* 札ボタン
    * 名前
    * 境界線
    * 効果
        * トークン
            * 変数操作
            * 特性表記

HTMLで記述すると以下。これは各種の概念を各字に翻訳し、更に改行を考慮した構造としてトークン単位に分けて構造化せねばならないことを意味する。実装量が膨れ上がってしまう。開発方針に反するため、これを却下とする。

```html
<button style="
    --btn-border: #666;
    width: 80px; height: 71px; padding: 2px 0;
    border: 1px solid var(--btn-border);
    display: flex; flex-direction: column;
    font-family: monospace; line-height: 1.0; background: #eee;
">
    <!-- 名前：18px -->
    <div style="
        height: 18px; display: flex; align-items: center; justify-content: center;
        font-size: 18px; font-weight: bold; width: 100%;
        border-bottom: 1px solid var(--btn-border);
    ">斬る</div>
    
    <!-- 要約：16px × 3行 (flex-wrapを使い、入り切らないトークンを次行へ送る) -->
    <div style="
        flex: 1; display: flex; flex-wrap: wrap; justify-content: center;
        font-size: 16px; width: 100%; letter-spacing: 0.05em;
    ">
        <span style="display: inline-block; white-space: nowrap;">敵♥-1</span>
        <span style="display: inline-block; white-space: nowrap;">♡-N</span>
        <span style="display: inline-block; white-space: nowrap;">[連斬]</span>
    </div>
</button>
```

効果要約は表示しない。札の説明は詳細によって行う。札ボタンには名前と状態のみを記述する。

#### 効果詳細

札の効果を説明する方法は、画面設計と用語定義に基づき、不要な文を極力削除した必要最小限の文章で表現する。その結果が以下である。

札名|詳細
----|----
`薬`|`自分の生命点を3増やす。`
`斬る`|`相手の生命点と身体点を1減らす。もしこの札を続けて選択したなら、身体点の減少量を1増やす(最大2)。さもなくば、減少量を0に戻す。`
`突貫`|`相手の生命点を3減らす。自分の生命点を1減らす。`
`自然治癒`|`自分の生命点を1増やす。この札は選択できず自動発動する。`

画面設計はscreen-design.md、用語の統一は用語集glossary.mdを閲覧すること。それらを前提として適切な文言を設計する。

240x320という限られた画面サイズで説明せねばならない。故に画面は有効活用すべきだ。必要最小限の文章を構築すべきである。また、用語や表記は統一し、表記ゆれをなくすべきである。

##### 文章構築規則

画面構成から自明の内容を削り、必要最小限の説明文を構築する。その論理をここにまとめる。

1. 冗長な全文を構築する
2. 必要最小限に削る
3. 意味が通じることを確認する

###### 1. 冗長な全文を構築する

作中最長の説明となる`斬る`の札を例に挙げる。

```
このゲームにおいて、この札の使用者は、この札を使用した時、次のような効果を発揮する。

この札の使用者は、相手の生命点を1減らしたあと、相手の身体点を1減らす。もしこの札の使用者がこの札を次の順でも続けて選択したなら、相手の身体点を減らす量を1増やす。但し最大2までを上限とする。もしこの札の使用者がこの札を次の順でも続けて選択しなかったなら、相手の身体点を減らす量は0に戻る。
```

###### 2. 必要最小限に削る

以下の内容を削る。

```
このゲームにおいて、この札の使用者は、この札を使用した時、次のような効果を発揮する。
この札の使用者は、
```

理由はプレイヤにとって自明だから。このゲームにおける範囲内の話であることはアプリというシステム境界により自明だ。この札の使用者が、使用時に効果を発揮することは、札ボタンというUI提示や、ボタン押下後に生命点が減るなどの変化により察せられる。また、この文章が札説明であることは、札ボタンにフォーカスが当たった時に説明文が表示されることから自明である。故に上記の上位文脈を説明する文章はすべて省略可能だ。

残る文章は以下である。これらも省略可能な箇所が多くある。

```
相手の生命点を1減らしたあと、相手の身体点を1減らす。もしこの札の使用者がこの札を次の順でも続けて選択したなら、相手の身体点を減らす量を1増やす。但し最大2までを上限とする。もしこの札の使用者がこの札を次の順でも続けて選択しなかったなら、相手の身体点を減らす量は0に戻る。
```

1文目。生命点と身体点という異なる属性に対して一律1減らしている。`1減らす`を一つにまとめることができるはずだ。この際、順序が重要だ。逆だと全く異なる結果になってしまう。但し順序に関しては説明文で記述した通りとすれば自明だ。すなわち、`相手の生命点と身体点を1減らす`と記述すれば、生命点を減らしてから身体点を減らすという順序を示せる。

2文目。`この札の使用者が`は省略可能は上位文脈のため削除できる。`次の順でも続けて選択したなら`は、ターン制バトルであるというゲームルールにおいて、順をまたいで続けて選択することは自明である。よって`次の順でも`は省略できる。

2文目の後半。`身体点を減らす量`については用語集により`減少量`を使う。これは後述の`減らす量を1増やす。`という表記が紛らわしいからだ。量に関しては用語集で`減少量`／`増加量`という名詞を使うことで絶対値を操作する動詞の`減らす`／`増やす`と差別化を図る。これにて少しだけ紛らわしさを誤魔化す。すなわち`身体点の減少量を1増やす。`にする。

3文目。`但し最大2までを上限とする。`は2文目の詳細だ。これは条件が真だった時の内容の一部であるため、できれば同一文内に収めたい。`最大`,`まで`,`上限`の3語は意味が重複しているため一つに絞れるだろう。これらをすべて実現すると、`減少量を1増やす(最大2)。`となる。

4文目。これは2文目の条件が偽だった場合だ。それは`if else`句の構文`もし〜なら、〜する。さもなくば、〜する。`で表記すべきである。すなわち`さもなくば`を使う。また、`相手の身体点を減らす量`は、この札効果において減らす量は相手の身体点しか存在しないことから、単に`減らす量`で済む。そしてそれを用語集の`減少量`で表現する。これらをすべて適用すると、`さもなくば、減少量を0に戻す。`になる。

###### 3. 意味が通じることを確認する

上記すべてを適用すると、用語集にある通り以下の文章になる。

```
相手の生命点と身体点を1減らす。もしこの札を続けて選択したなら、身体点の減少量を1増やす(最大2)。さもなくば、減少量を0に戻す。
```

これ以上は削れない。1文目の`相手のは削れない。削れば`自分の`か両者共だと誤解されかねないからだ。

`2文目後半の`身体点の`は省略できない。1文目では生命点と身体点を共に`1減らす`と表記しているため、どちらの属性に言及しているかは`減少`という動詞だけでは共通しており不明だ。減らす属性が何であるかを明示せねば、生命点・身体点の両方を操作すると解釈されてしまうだろう。よって`身体点の`は省略できない。但し2文目の`相手の`は省略できる。1文目の文脈を引き継いでいるからだ。

3文目は`相手の`も`身体点の`も省略できる。それ以前の文で言及され文脈を引き継いでいるからだ。

これ以上削ると意味が通じなくなるか誤解されてしまう。よって上記文が必要最小限の説明文であると説明できる。
